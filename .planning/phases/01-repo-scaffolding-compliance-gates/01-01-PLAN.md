<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm registry → root devDependencies | `pnpm install` pulls `turbo`, `typescript`, `eslint`, `typescript-eslint`, `license-checker-rseidelsohn`, `generate-license-file` from the public npm registry — untrusted code enters the build environment here |
| Corepack → pnpm binary | `corepack use pnpm@11.10.0` resolves and executes the pnpm binary itself from the npm registry |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Tampering | `pnpm install` (root devDependencies) | high | mitigate | pnpm v10+ blocks all `preinstall`/`postinstall` lifecycle scripts by default ("Security by Default"); `strictDepBuilds: true` + `onlyBuiltDependencies: []` in `pnpm-workspace.yaml` makes any *new* dependency needing a lifecycle script fail loudly instead of silently skipping, forcing an explicit reviewed allow-list decision. Confirmed via RESEARCH.md: none of this phase's packages declare a `postinstall` script. |
| T-01-02 | Tampering | Corepack pnpm version resolution | medium | mitigate | `packageManager: "pnpm@11.10.0"` pins an exact version (not a range); Task 1's checkpoint requires human confirmation on npmjs.com before this version is trusted and installed |
| T-01-03 | Repudiation | `pnpm-lock.yaml` drift between contributors | low | accept | Lockfile is committed; `--frozen-lockfile` enforcement in CI is wired in Plan 01-02, not this plan — accepted here since this plan only establishes the lockfile, doesn't yet run it in CI |
</threat_model>
